## 1. Метаданные и Экспресс-диагностика

* **Тип обращения:** Сбой синхронизации / Ошибка интеграции с KNX.
* **Симптом:** На физических настенных панелях KNX отображается некорректное системное время (сброс на `00:16` и дату `1. Янв. 1900`). Телеграммы времени некорректно обрабатываются шиной, либо в логах iRidi Server фиксируется отправка невалидных массивов.
* **Диагностический лог сервера:** 
  `[29-05-2026 14:36:26.468] INFO SCRIPT Sending time: 174,36,26`

---

## 2. Описание проблемы (Context)

При реализации синхронизации времени с шиной KNX системные интеграторы часто используют устаревшие методы ручной низкоуровневой упаковки даты и времени. В скриптах это выглядит как битовое смещение дня недели и часов для ручного формирования 3-байтового массива данных:

```javascript
var time = [(day << 5) | d.getHours(), d.getMinutes(), d.getSeconds()];
```

Такой подход приводит к отправке сырого массива вида `[174, 36, 26]` через метод `.Set()`. Однако встроенный драйвер KNX в iRidi Server при установленном типе данных `10.Time (10.001)` ожидает на вход стандартный JavaScript-объект `Date`. Попытка передать массив вместо объекта `Date` приводит к некорректной интерпретации драйвером, отправке невалидных данных в групповой адрес `7/0/0` и, как следствие, сбросу часов на оконечных устройствах KNX в дефолтное состояние (`00:16 / 1900-01-01`).

## 4. Официальное решение (Root Cause Fix)

Для настройки корректной автоматической синхронизации времени и даты выполните следующие действия:

### 1. Настройка типов данных в iRidi Studio
* Перейдите в дерево устройств сервера (`Devices`), найдите ваш драйвер KNX (например, `KNX IP Router`).
* Для группового адреса времени (например, `7/0/0`) установите:
  * **Type:** `10.Time`
  * **Sub Type:** `10.001 Time`
  * **Command type:** `Write value`
* Для группового адреса даты (например, `7/0/1`) установите:
  * **Type:** `11.Date`
  * **Sub Type:** `11.001 Date`
  * **Command type:** `Write value`

### 2. Корректировка скрипта в iRidi Studio
* Полностью удалите ручной расчет байтов и битовые сдвиги.
* Передавайте в метод `.Set()` нативный объект JavaScript `new Date()` напрямую. Драйвер iRidi автоматически упакует его в форматы DPT 10.001 и DPT 11.001 в зависимости от типа канала.

### 3. Оптимизация интервалов отправки
* Избегайте слишком частой отправки данных (например, раз в несколько секунд), так как это перегружает шину KNX.
* Рекомендуется установить интервал отправки в **60000 мс** (1 минута) или использовать событие перехода драйвера в онлайн.

### 4. Проверка кодировки файлов
* Убедитесь, что файлы JS-скриптов сохранены в кодировке **UTF-8 без BOM** для предотвращения появления некорректных символов в логах.

## 5. Технический Workaround

**Логика:** Выполнение первичной синхронизации через 10 секунд после старта сервера (время, необходимое для гарантированного перехода драйвера KNX в статус `ONLINE`) с последующей циклической отправкой данных каждую минуту.

```javascript
IR.AddListener(IR.EVENT_START, 0, function() {
   var KNX_DEVICE = IR.GetDevice("KNX IP Router");
   var CH_DATE = "7/0/1"; // Групповой адрес для Даты (DPT 11.001)
   var CH_TIME = "7/0/0"; // Групповой адрес для Времени (DPT 10.001)

   function syncDateTime() {
      if (KNX_DEVICE) {
         var currentDate = new Date();
         // Нативная автоматическая конвертация силами iRidi Server:
         KNX_DEVICE.Set(CH_DATE, currentDate);
         KNX_DEVICE.Set(CH_TIME, currentDate);
      }
   }

   // Первичная синхронизация с задержкой 10 сек после старта системы
   IR.SetTimeout(10000, function() {
      syncDateTime();
   });

   // Интервал синхронизации - 1 раз в минуту
   IR.SetInterval(60000, function() {
      syncDateTime();
   });
});
```

Так как в настройках группового адреса `7/0/0` в iRidi Studio указан тип данных `10.Time (10.001)`, драйвер iRidi пытается самостоятельно обработать отправляемое значение. Передача сырого массива байтов вместо встроенного JS-объекта `Date` вызывает ошибку конвертации, из-за чего в шину уходят невалидные данные, приводящие к сбросу времени на панелях к значению по умолчанию (`00:16 1. Янв. 1900`).

> **Решение:**
> 1. Проверьте, что в свойствах команды `7/0/0 - ВРЕМЯ` в iRidi Studio установлены параметры: **Type** = `10.Time`, **Sub Type** = `10.001 Time`, **Command type** = `Write value`.
> 2. Замените код в скрипте `CLOCK` на отправку стандартного JS-объекта даты. Драйвер iRidi автоматически преобразует его в нужный формат KNX.
> 3. Снизьте частоту отправки до 60 секунд, чтобы избежать избыточной нагрузки на шину KNX.

## 💡 Ключевые Инсайты (NotebookLM)
1. **Автоматизация упаковки данных iRidi:** Современный драйвер iRidi Server самостоятельно обрабатывает нативные объекты JavaScript `Date` и транслирует их в форматы DPT 10.001 (время) и DPT 11.001 (дата), исключая необходимость ручной сборки массивов разработчиком.
2. **Устаревшие практики в скриптах:** Попытка передачи собранного вручную 3-байтового массива типа `[174, 36, 26]` в канал с типом данных `10.Time` распознается драйвером как некорректная структура, что ведет к отправке в групповой адрес `7/0/0` нулевых значений и сбросу времени на дисплеях панелей KNX.
3. **Оптимизация трафика шины:** Чрезмерно частая запись времени (например, без фильтрации или со слишком малым интервалом) перегружает шину KNX паразитными телеграммами; стандартным и безопасным решением является синхронизация раз в 60 секунд с предварительной отсечкой времени инициализации шлюза при старте.
