### Описание проблемы

Пользователи могут сталкиваться с нестабильной работой серверов (в частности, ProAV Basic и решений на базе Raspberry Pi), которая проявляется следующими симптомами:
* Периодическая самопроизвольная перезагрузка устройства.
* Сброс сервисного пароля и сетевых настроек.
* Удаление загруженного проекта из памяти.
* Невозможность войти в веб-интерфейс или подключиться по SSH.
* Ошибка `Segmentation Fault` в логах при запуске сервера.

### Первичная диагностика

При возникновении подобных симптомов необходимо проверить наличие crash-файла, который блокирует нормальную работу службы.

1. Подключитесь к серверу по SSH.
2. Проверьте наличие файла `/iridiumserver/ircr.txt`. Если он существует, сервер будет пытаться считать его при каждой загрузке, что может приводить к циклической ошибке.
3. Удалите файл командой:
   ```bash
   rm /iridiumserver/ircr.txt
   ```
4. Перезагрузите сервер и проверьте стабильность работы.

### Проверка файловой системы

Проблема может быть вызвана отсутствием свободного места на диске.
1. Выполните команду `df -h`.
2. Если раздел `/overlay` или корневой раздел заполнены на 100%, необходимо очистить логи или старые версии проектов.

### Аппаратная диагностика памяти (EMMC)

Одной из наиболее вероятных причин «сброса» настроек является переход внутренней памяти в режим «только чтение» (Permanent Write Protect) из-за износа или брака конкретных партий модулей.

#### 1. Идентификация модуля памяти
Выполните команду для определения производителя памяти:
```bash
dmesg | grep mmc1
```
Если в выводе фигурирует модель **TLi** (например, `mmcblk1: mmc1:0001 TLi16G`), такое устройство может требовать замены по гарантии при подтверждении ошибок записи.

#### 2. Поиск системных ошибок ввода-вывода
Просмотрите последние сообщения ядра:
```bash
dmesg | tail -n 50
```
Критическими признаками неисправности являются строки:
* `Buffer I/O error on dev mmcblk...` — физический сбой записи.
* `print_req_error: I/O error` — деградация секторов памяти.
* `timed out waiting for interrupt` — зависание контроллера памяти.

#### 3. Тест на блокировку (Read-only)
Проверьте, позволяет ли система создавать файлы:
```bash
touch /overlay/test_write && rm /overlay/test_write
```
Если консоль возвращает ошибку `Read-only file system`, значит, чип памяти аппаратно заблокирован для предотвращения потери данных и требует замены (RMA).

### Рекомендуемые действия

1. **Откат версии:** Если проблема появилась после обновления ПО, выполните откат на стабильную версию через веб-интерфейс или SSH (инструкции в разделе «Обновление и откат версии»).
2. **Сброс к заводским настройкам:** Если аппаратные ошибки не обнаружены, выполните полный сброс устройства для исключения логических ошибок файловой системы.
3. **Удаленная диагностика:** Если самостоятельная проверка не выявила причину, подготовьте доступ через **RuDesktop** для технической поддержки (AnyDesk может блокироваться).
4. **Замена устройства:** При обнаружении ошибок `I/O error` и наличии модуля памяти TLi обратитесь в сервисный центр для замены SOM-модуля.
