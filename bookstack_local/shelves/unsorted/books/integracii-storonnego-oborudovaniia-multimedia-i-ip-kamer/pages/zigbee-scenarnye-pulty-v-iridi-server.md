## Описание

На текущий момент Zigbee-сценарные пульты не поддерживаются штатным Zigbee-драйвером iRidi Server в полном объёме.

К таким устройствам относятся Zigbee-пульты, которые не управляют нагрузкой напрямую, а передают события нажатий кнопок, например:

- одиночное нажатие;
- двойное нажатие;
- длительное удержание.

Такие устройства часто используются для запуска пользовательских сценариев: включения света, выбора сцены, управления реле, шторами, климатом или другими устройствами автоматизации.

## Особенность работы сценарных Zigbee-пультов

В отличие от обычных реле, диммеров или датчиков, сценарный пульт не всегда передаёт состояние в виде стандартного значения `ON` / `OFF`.

Чаще всего он отправляет событие действия, например:

```text
1_single
1_double
1_hold
2_single
2_double
2_hold
```

Для корректной работы такого устройства сервер должен не только принять Zigbee-сообщение, но и правильно интерпретировать его как событие кнопки, после чего передать это событие в проект в виде доступного feedback-тега.

## Текущее ограничение

Штатный Zigbee-драйвер iRidi Server может определить сценарный пульт как Zigbee-устройство и получать от него входящие сообщения на уровне Zigbee-сети.

Однако такие сообщения на текущий момент могут не преобразовываться в стабильные события тегов, доступные для обработки в проекте.

Из-за этого возможны следующие ситуации:

- пульт отображается в списке Zigbee-устройств;
- устройство импортируется в проект неполно или некорректно;
- события кнопок не появляются в проекте как отдельные feedback-теги;
- нажатия кнопок видны в логах сервера, но не обрабатываются в пользовательском сценарии;
- события могут определяться нестабильно или только частично.

Это означает, что использовать сценарные Zigbee-пульты напрямую через штатный Zigbee-драйвер iRidi Server на данный момент не рекомендуется.

## Рекомендуемый обходной вариант

В качестве обходного решения можно использовать внешний Zigbee-шлюз на базе Zigbee2MQTT и передавать события пульта в iRidi Server через MQTT.

Рекомендуемая схема:

```text
Zigbee-сценарный пульт
        ↓
Zigbee-координатор
        ↓
Zigbee2MQTT
        ↓
MQTT-брокер
        ↓
MQTT-драйвер iRidi Server
        ↓
Серверный проект iRidi
        ↓
Сценарии / управление устройствами
```

В этой схеме Zigbee2MQTT отвечает за работу с Zigbee-сетью и корректную интерпретацию событий сценарного пульта. iRidi Server получает уже готовые MQTT-сообщения и обрабатывает их в серверном проекте.

## Что потребуется для реализации обходного варианта

Для реализации такого решения потребуется:

- Raspberry Pi, мини-ПК или другой сервер, на котором можно запустить Zigbee2MQTT;
- Zigbee-координатор, совместимый с Zigbee2MQTT;
- MQTT-брокер, например Mosquitto;
- Zigbee2MQTT;
- MQTT-драйвер в проекте iRidi Server;
- серверный скрипт iRidi для обработки входящих MQTT-сообщений.

## Пример MQTT-события от сценарного пульта

После добавления пульта в Zigbee2MQTT его события обычно публикуются в MQTT-топик устройства.

Пример топика:

```text
zigbee2mqtt/scenario
```

Пример payload:

```json
{
  "action": "1_single",
  "battery": 100,
  "linkquality": 120
}
```

В этом примере основное значение для обработки — поле `action`.

Именно по нему можно определить, какая кнопка была нажата и какой тип нажатия был выполнен.

## Пример обработки событий

В проекте iRidi Server можно подписаться на MQTT-топик сценарного пульта и обработать значение `action` в JavaScript.

Пример логики:

```javascript
function handleScenarioRemote(payload) {
    var data;

    try {
        data = JSON.parse(payload);
    } catch (e) {
        IR.Log("MQTT parse error: " + e);
        return;
    }

    if (!data.action) {
        return;
    }

    IR.Log("Scenario remote action: " + data.action);

    switch (data.action) {
        case "1_single":
            // Выполнить сценарий для одиночного нажатия кнопки 1
            break;

        case "1_double":
            // Выполнить сценарий для двойного нажатия кнопки 1
            break;

        case "1_hold":
            // Выполнить сценарий для удержания кнопки 1
            break;

        case "2_single":
            // Выполнить сценарий для одиночного нажатия кнопки 2
            break;
    }
}
```

Конкретная реализация зависит от структуры проекта, названий драйверов, команд и feedback-тегов.

## Пример управления Zigbee-реле через MQTT

Если через Zigbee2MQTT также добавлено Zigbee-реле, им можно управлять из iRidi Server через MQTT.

Пример топика управления:

```text
zigbee2mqtt/relay/set
```

Payload для включения:

```json
{
  "state": "ON"
}
```

Payload для выключения:

```json
{
  "state": "OFF"
}
```

Пример топика фидбека:

```text
zigbee2mqtt/relay
```

Пример payload фидбека:

```json
{
  "state": "ON",
  "linkquality": 130
}
```

Таким образом, через MQTT можно не только получать события от сценарного пульта, но и управлять другими Zigbee-устройствами.

## Важные ограничения обходного решения

При использовании Zigbee2MQTT необходимо учитывать следующие ограничения:

- Zigbee-координатор должен использоваться только одним сервисом одновременно.
- Если координатор уже используется iRidi Server, Zigbee2MQTT не сможет работать с ним параллельно.
- Если координатор используется Zigbee2MQTT, штатный Zigbee-драйвер iRidi Server не должен обращаться к этому же координатору.
- После изменения схемы работы может потребоваться повторное добавление Zigbee-устройств в сеть Zigbee2MQTT.
- Логику сценариев необходимо реализовать в серверном проекте iRidi через MQTT-драйвер и JavaScript.

## Рекомендация

Если в проекте требуется использовать Zigbee-сценарные пульты, рекомендуется применять связку Zigbee2MQTT + MQTT-драйвер iRidi Server.

Штатное подключение таких пультов напрямую через Zigbee-драйвер iRidi Server на данный момент не рекомендуется, так как события кнопок могут не преобразовываться в корректные feedback-теги проекта.

## Статус поддержки

Сценарные Zigbee-пульты на данный момент следует считать неподдерживаемыми штатным Zigbee-драйвером iRidi Server.

Поддержка таких устройств может быть рассмотрена в будущих версиях продукта.
