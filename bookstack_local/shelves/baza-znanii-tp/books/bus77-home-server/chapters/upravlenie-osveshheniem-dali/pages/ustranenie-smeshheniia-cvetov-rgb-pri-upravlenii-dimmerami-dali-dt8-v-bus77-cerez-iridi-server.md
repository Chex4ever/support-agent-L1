## 1. Метаданные и Экспресс-диагностика
* **Тип обращения:** Сбой цветопередачи RGB / Проблема интеграции драйвера Bus77 и DALI DT8
* **Симптом:** Цвета на светодиодной ленте DALI DT8 не соответствуют выбранным на палитре ColorPicker (наблюдается выраженное смещение в сине-фиолетовый спектр при попытке установить теплые тона). В веб-интерфейсе iRidi Server во вкладке Feedbacks для RGB-тега отображаются некорректные отрицательные значения (например, `-62970369.000000` или `-2076926977.000000`).
* **Статус объекта:** ПНР / Эксплуатация

## 2. Описание проблемы (Context)
При интеграции диммеров DALI DT8 (в частности, *INTELLIGENT ARLIGHT SMART-DALI-104-72-SH*) с iRidi Server через шлюз и драйвер Bus77 возникает конфликт разрядности и структуры передаваемых данных. Графический элемент управления `ColorPicker` в приложении iRidi по умолчанию генерирует и передает цвет в 32-битном формате `RRGGBBAA` (где `AA` — это младший байт, отвечающий за прозрачность/альфа-канал).

Драйвер `Bus77` для управления цветовыми каналами DALI DT8 ожидает строго 24-битное значение в формате `RRGGBB`. При прямой передаче значения без предварительной обработки байт прозрачности `AA` смещает всю последовательность бит. В результате драйвер неверно интерпретирует цветовые каналы (происходит наложение синего спектра и альфа-канала на красный и зеленый), что приводит к визуальному искажению свечения ленты. Появление отрицательных чисел в панели сервера вызвано тем, что беззнаковое 32-битное число RGBA (где старший бит равен 1) ошибочно интерпретируется веб-интерфейсом как знаковое 32-битное целое (Signed Int32). Это является исключительно косметическим артефактом отображения и не влияет на физический уровень управления при условии корректного форматирования данных.

## 4. Официальное решение (Root Cause Fix)
Для восстановления корректной цветопередачи необходимо отсечь младшие 8 бит (байт прозрачности `AA`) перед отправкой значения в шину Bus77. В среде iRidi Studio это реализуется через визуальную схему логики:

1. Откройте ваш проект в **iRidi Studio** и перейдите во вкладку редактора схем автоматизации (**Scripts / Blocks**).
2. Добавьте на рабочую область блок побитового сдвига вправо — **RSHIFT** (Right Shift).
3. Свяжите входной параметр **Value** блока `RSHIFT` со статусным тегом, получающим данные от графического элемента `ColorPicker` (формат `RRGGBBAA`).
4. Для параметра **Count** (величина сдвига в битах) задайте константу со значением `8`.
5. Выходной разъем блока `RSHIFT` (сформированное 24-битное значение `0x00RRGGBB`) подключите к управляющему каналу (RGB Channel) драйвера **Bus77**.
6. Сохраните изменения, скомпилируйте проект и загрузите обновленную конфигурацию на iRidi Server.

## 5. Технический Workaround
Если логика проекта реализуется на уровне сценариев JavaScript, используйте оператор беззнакового сдвига вправо (`>>>`), который автоматически отсекает младший байт и исключает появление знакового бита:

```javascript
// Функция приведения 32-битного RGBA к 24-битному RGB
function convertRGBAtoRGB(rgbaValue) {
    // Беззнаковый сдвиг вправо на 8 бит отбрасывает байт прозрачности (AA)
    return rgbaValue >>> 8;
}

// Пример интеграции в обработчик событий iRidi
IR.AddListener(IR.EVENT_TAG_CHANGE, IR.GetDevice("UI").GetFeedback("ColorTag"), function(name, value) {
    // Получаем значение, выполняем побитовый сдвиг и пишем в шину
    var cleanRGB = value >>> 8;
    IR.GetDevice("BUS77").Set("RGB_Channel", cleanRGB);
});
```

## 6. Шаблон ответа клиенту
### RU
Приветствую!

Описанная вами ситуация со смещением цветов светодиодной ленты DALI DT8 и отображением отрицательных значений в панели iRidi Server обусловлена разницей форматов передачи данных:

1. **Искажение палитры:** Визуальный элемент `ColorPicker` в iRidi работает в 32-битном формате (`RRGGBBAA`), где последние 8 бит определяют прозрачность. Драйвер `Bus77` для диммеров DALI ожидает 24-битное значение (`RRGGBB`). Из-за избыточного байта происходит сдвиг фаз цветовых каналов, что уводит итоговое свечение в синий спектр.
2. **Отрицательные числа:** Значения вида `-62970369.000000` в веб-панели сервера — это визуальный артефакт интерпретации беззнакового 32-битного числа как знакового (Signed Int). На работу оборудования это не влияет, но подтверждает некорректный формат данных в шине.

**Решение:**
Перед отправкой переменной в канал Bus77 необходимо отсечь последние 8 бит. Для этого в редакторе логики **iRidi Studio** используйте блок побитового сдвига **RSHIFT**:
* Направьте значение тега `ColorPicker` на вход **Value** блока **RSHIFT**.
* В поле **Count** установите константу `8`.
* Результат сдвига свяжите с управляющему каналом диммера в драйвере **Bus77**.

Это полностью устранит искажение цветов и стабилизирует работу оборудования. Скриптовый вариант решения на JavaScript также прикреплен к данному обращению.

## 💡 Ключевые Инсайты (NotebookLM)
* **Разрядный конфликт стандартов:** Прямое связывание 32-битного значения `RRGGBBAA` от виджета ColorPicker iRidi с 24-битным драйвером Bus77 приводит к побитовому сдвигу, из-за которого байт прозрачности (`AA`) перезаписывает цветовые каналы, уводя свечение ленты в синий спектр.
* **Природа знакового артефакта:** Отрицательные числа (вплоть до `-2076926977.000000`) в веб-интерфейсе iRidi Server — это исключительно косметический баг парсинга данных на веб-странице, где беззнаковое 32-битное целое число (Unsigned Int32) ошибочно интерпретируется как знаковое (Signed Int32).
* **Аппаратная независимость решения:** Проблема решается полностью программным путем на стороне сервера iRidi (через блок `RSHIFT` с константой `8` или оператор `>>> 8` в JS), что исключает необходимость физической перекоммутации оборудования DALI или изменения настроек шлюза.
