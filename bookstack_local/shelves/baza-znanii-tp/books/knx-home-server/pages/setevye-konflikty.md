# Диагностика ошибок «Другой проект на сервере» и «Неправильный ключ лицензии» (KNX Home Server)

**Категория:** KNX Home Server / i3 KNX / Сетевые конфликты  
**Связанные продукты:** i3 KNX, KNX Home Server  
**Обновлено:** 03.07.2026 (на основе Postmortem Аварии #16662)  
**Источник:** тикет Omnidesk #443-532569, Redmine #16662

---

## 1. Описание проблемы (симптомы)

При работе с приложением **i3 KNX** и серверами **KNX Home Server** клиент может столкнуться со следующими симптомами:

* Ошибка **«Другой проект на сервере»** при загрузке проекта (иногда проходит после нескольких попыток).
* Ошибка **«Неправильный ключ лицензии»** при попытке восстановить сервер через облако.
* Ошибка **«Error receive server response»** при попытке восстановить сервер по локальной сети (особенно если сервер подключен через VPN).
* Самопроизвольная смена IP-адреса в настройках проекта i3 KNX (например, IP меняется с `192.168.0.160` на `192.168.1.113`).
* Появление дубликатов файлов проектов на ПК интегратора.
* Сброс до заводских настроек не устраняет проблему.

> **Важно:** Клиенты часто ошибочно полагают, что кнопка **«Сменить сервер»** в конфигураторе i3 KNX предназначена для переключения между *активными* серверами в одной сети. На самом деле она предназначена для переноса проекта на *другое физическое устройство*. Нажатие этой кнопки в описанной конфигурации гарантированно приводит к подмене HWID в файлах проекта.

---

## 2. Root Cause (первопричина)

**RC-CONF — нестабильная пользовательская конфигурация.**

Проблема возникает при наличии в одной подсети (или через VPN) нескольких серверов **KNX Home Server** без жёсткого резервирования IP-адресов. Из-за сетевых конфликтов или особенностей маршрутизации VPN приложение i3 KNX на ПК интегратора «видит» не тот сервер, подменяет IP-адрес и HWID в файлах проекта, что приводит к повреждению привязок и созданию дублей файлов.

**Ключевые факторы:**

1. Два и более KNX Home Server в одной подсети / через VPN.
2. Отсутствие статических IP или резервирования по MAC в DHCP.
3. Ручное нажатие кнопки «Сменить сервер» в i3 KNX.
4. Ручное редактирование файлов проектов (в т.ч. через WinSCP/SSH).

> На активированном сервере в директории `/var/lib/iRidium Server KNX/Documents/` должен находиться **строго один проект**. Наличие дублей — признак конфликта привязки.

---

## 3. Чек-лист диагностики (Preventive Action)

При поступлении обращения с вышеуказанными симптомами инженер **L1/L2 обязан** выполнить следующие шаги **до** эскалации на разработку.

### Шаг 1. Опрос клиента (сбор конфигурации)

Уточнить у клиента следующие параметры:

1. **Топология сети:** сколько KNX Home Server находится в одной локальной сети / подсети клиента?
2. **IP-адресация:** настроено ли резервирование IP-адресов (DHCP Static Lease) или заданы ли статические IP-адреса для обоих серверов на роутере?
   * *Если нет — рекомендовать настроить в первую очередь.*
3. **Наличие VPN:** подключён ли один из серверов через VPN? (VPN может создавать конфликты маршрутизации.)
4. **Действия клиента:**
   * Нажимал ли клиент кнопку **«Сменить сервер»** в конфигураторе i3 KNX?
   * Пытался ли он вручную редактировать файлы проектов?
   * Выполнялся ли сброс до заводских настроек?
5. **Проекты в облаке:**
   * Наименования проектов.
   * Логин учётной записи, с которой они созданы.
   * Ключи лицензий и соответствие «ключ → проект».

### Шаг 2. Удалённая диагностика

Если Шаг 1 не дал результата (IP зарезервированы, но ошибка сохраняется), запросить удалённое подключение (**RuDesktop / AnyDesk**) к ПК клиента:

1. Проверить наличие дубликатов файлов проектов на ПК (например, два файла `HOME` и два файла `TEST`).
2. Открыть файлы проектов и сравнить прописанный внутри **HWID** с реальным HWID сервера (HWID сервера можно посмотреть в веб-интерфейсе или через SSH).
3. **Критерий ошибки:** если в файле проекта HWID не совпадает с реальным HWID целевого сервера, или если в файлах разных проектов прописан одинаковый HWID — это подтверждает конфликт привязки.

### Шаг 3. Проверка сетевой доступности

1. Проверить пинги до серверов из ПК клиента.
2. Убедиться, что приложение i3 KNX «видит» именно тот сервер, к которому привязан проект (а не соседний в той же подсети).
3. При наличии VPN — проверить, не происходит ли подмена маршрутов.

## 4. Алгоритм решения (Workaround)

Если подтверждён конфликт HWID/IP в файлах проектов, выполняются следующие действия **удалённо**:

1. Вручную прописать корректные HWID в файлы проектов на ПК клиента.
2. Удалить все лишние дубликаты файлов проектов с ПК.
3. Очистить кэш приложения i3 KNX.
4. Сохранить исправленные проекты в облако iRidium.
5. Проверить загрузку проекта на сервер.
6. Проверить одновременную работу обоих серверов (если конфигурация этого требует).

> **Важно по лицензиям:** ключ лицензии с наклейки на корпусе привязывается к проекту i3 KNX. При замене сервера необходимо учитывать:
> * Новый, ранее не добавленный ни в один проект сервер — ключ с наклейки можно использовать в связке с другим сервером.
> * Сервер, на котором уже есть проект, — необходимо предварительно деактивировать его; ключ с такого сервера при замене не подойдёт.

**Процедура деактивации сервера в i3 KNX:**

1. Выйти из проекта, если он открыт.
2. Нажать кнопку **«Новый проект»**.
3. Найти и выбрать сервер для деактивации из списка доступных.
4. В окне «Сервер активирован в другом проекте» нажать **«Отключить сервер»**.
5. В окне «Внимание» нажать **«Отключить»**.
6. Ввести пароль аккаунта, которым сервер был активирован, и нажать **«ОК»**.
7. Дождаться завершения процедуры деактивации.

## 5. Эскалация на L2 / Отдел разработки

Если ручная корректировка HWID, удаление дублей и настройка статических IP-адресов **не устранили проблему**, необходимо:

1. Собрать логи с проблемного сервера через SSH:
   * `/var/log/messages`
   * Файлы из `/var/lib/iRidium Server KNX/`
2. Запросить у клиента файлы повреждённых проектов.
3. Передать обращение на **L2 (Отдел разработки i3 KNX)** с пометкой, что стандартные меры по устранению **RC-CONF** (статические IP, ручная правка HWID) не помогли.
4. Приложить к задаче в Redmine:
   * Ссылку на тикет Omnidesk.
   * Логи, файлы проектов, скриншоты.
   * Описание конфигурации клиента (количество серверов, VPN, IP-адреса).
