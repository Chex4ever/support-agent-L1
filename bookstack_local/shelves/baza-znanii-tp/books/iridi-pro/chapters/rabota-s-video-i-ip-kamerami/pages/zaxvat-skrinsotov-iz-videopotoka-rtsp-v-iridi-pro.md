# Захват скриншотов из видеопотока RTSP

В текущих версиях iRidi Pro прямой захват кадра (скриншота) непосредственно из активного RTSP-видеопотока стандартными средствами визуализации не поддерживается. Для реализации функционала отображения снимка с камеры необходимо использовать альтернативные методы.

## Способы получения изображения

### 1. Использование Snapshot URL
Наиболее эффективный и менее ресурсозатратный способ. Большинство современных IP-камер поддерживают получение статического изображения (JPEG) по прямому HTTP-запросу. 
*   **Преимущество:** Минимальная нагрузка на контроллер и сеть.
*   **Реализация:** Настройте получение данных по ссылке вида `http://[IP-адрес]/snapshot.jpg` (синтаксис зависит от производителя камеры).

### 2. Метод IR.CreateImageFromBuffer
Если камера не поддерживает Snapshot URL или требуется обработка кадра на стороне, можно использовать JS API для создания изображения из буфера данных в формате base64.

**Синтаксис:**
```javascript
IR.CreateImageFromBuffer(name, {base64});
```

**Пример реализации:**
Для обновления изображения на панели управления используйте следующий алгоритм:
1. Получите строку base64 от внешнего источника.
2. Вызовите метод создания изображения.
3. Присвойте созданное изображение графическому элементу.

```javascript
var base64Data = "iVBORw0KGgoAAAANSUh..."; // Данные кадра
var imgId = IR.CreateImageFromBuffer("CameraFrame_1", base64Data);

// Отображение на странице
IR.GetPage("Page 1").GetItem("Item 1").GetState(0).Image = IR.GetImage(imgId);
```

## Интеграция через внешний сервис

Поскольку iRidi Pro не выполняет дешифровку видеопотока для захвата кадров, рекомендуется настроить промежуточный сервис (например, скрипт на Python или Node.js), который будет:
1. Подключаться к RTSP-потоку через ffmpeg.
2. Извлекать кадр с заданной периодичностью.
3. Кодировать кадр в base64 и передавать его в iRidium через HTTP, MQTT или WebSocket.

### Рекомендации по оптимизации:
*   **Частота обновления:** Ограничьте передачу кадров до 1–2 FPS во избежание перегрузки интерфейса.
*   **Сжатие:** Предварительно сжимайте JPEG перед кодированием в base64.
*   **Объем данных:** Помните, что base64 увеличивает размер передаваемых данных примерно на 33%.

## Аппаратные ограничения

Не рекомендуется использовать iRidi Home Server (HSS) или BUS77 Home Server для задач тяжелого видеопроцессинга (например, запуск ffmpeg для разбора потоков). 
*   Серверы оптимизированы для задач автоматизации и логики управления.
*   На устройствах отсутствует менеджер пакетов для установки стороннего ПО видеообработки.
*   Использование стороннего софта для видеопроцессинга на контроллерах может привести к нестабильности системы и аннулированию гарантийных обязательств.

Для стабильной работы системы выносите задачи обработки видео на выделенные серверные мощности.
