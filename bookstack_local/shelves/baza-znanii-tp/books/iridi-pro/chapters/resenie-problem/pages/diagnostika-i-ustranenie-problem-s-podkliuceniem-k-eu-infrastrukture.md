## 1. Метаданные и Экспресс-диагностика
* **Тип обращения:** Диагностика и устранение сетевых ограничений
* **Симптом:** Сбои синхронизации проекта, отсутствие авторизации, сервер не переходит в онлайн-статус.
* **Статус объекта:** ПНР / Эксплуатация

## 2. Описание проблемы (Context)
Серверы iRidium могут испытывать трудности с доступом к облачной инфраструктуре при нахождении объекта в зонах с активным сетевым фильтрованием (DPI, блокировка по SNI или IP). Проблема выражается в невозможности установить TLS-соединение с EU-хостами, что приводит к зависанию службы iRidi Server или отказам в авторизации. Настоящая инструкция позволяет дифференцировать тип блокировки и применить меры по локализации работы сервера в рамках RU-инфраструктуры.

## 3. Diagnostic Mapping (Symptom → Cause)
| Визуальный симптом / Ошибка в логах | Вероятная причина | Обязательные данные для сбора |
| :--- | :--- | :--- |
| Connection timed out / Refused | Блокировка IP-подсети (Hetzner) | Результат `ping` и `traceroute` |
| TLS Handshake failure (errno 104) | Блокировка SNI (DPI-фильтрация) | Лог `openssl s_client` |
| Connection hanging (no data) | Блокировка портов Cloud Gate | Результат `telnet` / `nc` |

## 4. Официальное решение (Root Cause Fix)
Для объектов, работающих в РФ, принудительно перенаправьте запросы на локальную инфраструктуру:
1. Отредактируйте файл `/etc/hosts` (Linux) или `C:\Windows\System32\drivers\etc\hosts` (Windows).
2. Добавьте записи для блокировки EU-хостов:
   `127.0.0.1 auth.eu.iridi.com`
   `127.0.0.1 i3pro.eu.iridi.com`
   `127.0.0.1 i3knx.eu.iridi.com`
   `127.0.0.1 bus77home.eu.iridi.com`
3. Удалите файл `irhosts.json` из директории документов сервера (`/var/lib/iRidium Server/Documents/`).
4. Перезапустите службу сервера или ОС.

## 5. Технический Workaround
* **Логика:** Выявление типа блокировки через анализ TLS-хендшейка. Если соединение сбрасывается сразу после Client Hello (SNI-фильтрация), необходимо использование VPN-туннеля для обхода ТСПУ.
* **Пример кода (анализ SNI):**
```bash
openssl s_client -connect auth.eu.iridi.com:443 -servername auth.eu.iridi.com
```
Если вывод содержит `write:errno=104`, значит, провайдер блокирует домен по SNI.

## 6. Шаблон ответа клиенту
Здравствуйте. Для устранения проблем с синхронизацией, пожалуйста, проведите диагностику сетевого соединения согласно инструкции. Если сервер не проходит проверку TLS (ошибка SNI), это свидетельствует о блокировках на уровне вашего интернет-провайдера. Рекомендуем проверить настройки межсетевого экрана или использовать альтернативный канал связи (VPN) для взаимодействия с облаком iRidium.

## 💡 Ключевые Инсайты (NotebookLM)
1. Ошибки типа `errno 104` при TLS-хендшейке являются прямым индикатором DPI-фильтрации по доменному имени (SNI).
2. Редактирование локального файла `hosts` позволяет принудительно исключить обращение сервера к заблокированным EU-узлам, стабилизируя работу локальных компонентов.
3. Использование инструментов `openssl` и `curl` позволяет достоверно отличить блокировку конкретного IP-адреса от блокировки по доменному признаку.
