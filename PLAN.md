# План внедрения AI-агента первой линии техподдержки iRidi

## 1. Проблема

- Нехватка специалистов первой линии
- Пользователи (интеграторы) дают мало информации, часто ошибочную
- Продукты имеют запутанные названия (iRidi vs iRidium, Bus77 Home ≠ железо и т.д.)
- Сотрудники тратят много времени на "угадывание" что у пользователя
- Нет единой системы подсказок для первой линии

## 2. Цель

Создать AI-агента, который:
1. Анализирует текст обращения пользователя
2. Определяет: продукт, категорию проблемы, недостающие данные
3. Ищет ответ в базе знаний (doc.iridi.com, dev.iridi.com, Omnidesk KB)
4. Оставляет заметку (suggestion/note) для сотрудника ТП с:
   - Рекомендуемым ответом
   - Ссылками на документацию
   - Какими вопросами нужно уточнить у пользователя
   - Уровнем уверенности (чтобы сотрудник понимал, стоит ли доверять)

## 3. Архитектура

```
┌──────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Omnidesk    │────▶│  support-agent   │────▶│  Knowledge Base │
│  (ticket)    │     │  (AI Agent)      │     │  (локальные MD) │
└──────────────┘     │                  │     └─────────────────┘
                     │  1. Классификация│     ┌─────────────────┐
                     │  2. Поиск ответа │────▶│  doc.iridi.com  │
                     │  3. Генерация    │     │  (веб-скрапинг) │
                     │     рекомендации │     └─────────────────┘
                     └──────────────────┘     ┌─────────────────┐
                                              │  Omnidesk KB    │
                                              │  (API)          │
                                              └─────────────────┘
```

### Компоненты

| Компонент | Назначение | Технология |
|-----------|-----------|------------|
| **Ticket Ingestion** | Получение тикета из Omnidesk (webhook/polling) | Python, Omnidesk API |
| **Classifier** | Определение продукта, категории, настроения | NLP + LLM |
| **Knowledge Retriever** | Поиск по базе знаний + веб | Vector search + Regex |
| **Response Generator** | Формирование заметки для сотрудника | LLM prompt |
| **KB Manager** | Управление базой знаний (CRUD) | Git + Markdown |

## 7. Omnidesk API Integration

### Основные эндпоинты
- `GET /api/v1/cases` — получение списка тикетов
- `GET /api/v1/cases/{id}` — получение конкретного тикета
- `POST /api/v1/cases/{id}/notes` — добавление заметки
- `POST /api/v1/cases/{id}/reply` — отправка ответа (для будущих фаз)

### Trigger: Webhook при создании тикета
Omnidesk поддерживает webhooks при:
- Создании тикета
- Обновлении тикета
- Добавлении сообщения

Формат payload — JSON с данными тикета.

## 9. Технический стек

- **Язык**: Python 3.14+
- **LLM**: Пока rule-based + embedding search; в будущем — любая LLM (через OpenRouter / локальная)
- **Vector store**: FAISS (локально) или Qdrant
- **API framework**: FastAPI (для вебхуков)
- **База знаний**: Markdown + YAML frontmatter
- **Поиск**: TF-IDF (начальный этап) → Embedding search
- **Git**: Версионирование базы знаний

## 10. Риски

| Риск | Вероятность | Смягчение |
|------|------------|-----------|
| Пользователи неверно указывают продукт | Высокая | Классификатор + уточняющие вопросы |
| Документация устарела или неполна | Средняя | Пометка confidence score |
| Ложные срабатывания (рекомендация неверного ответа) | Средняя | Режим "только заметка", не автоматический ответ |
| Сопротивление сотрудников | Низкая | Постепенное внедрение, учет фидбека |
| Изменения в API Omnidesk | Низкая | Изоляция слоя API |
